# what-is-this

파일 또는 함수(메서드)를 받아 정해진 순서로 설명하는 스킬. 목적은 "이 코드가 무엇이고, 어떻게 동작하며, 무엇을 조심해야 하고, 어떻게 개선할 수 있는가"를 한눈에 파악하게 하는 것이다.

## 대상 파악

먼저 무엇을 설명할지 확정한다.

- **파일**을 받았으면: 파일 전체를 Read로 읽는다. 300줄이 넘으면 핵심 부분(클래스 선언, public 메서드, 주요 필드)을 우선 파악한다.
- **함수/메서드**를 받았으면: 그 함수가 정의된 파일을 읽되, 함수 본문과 그것이 호출하는 주요 협력 객체(주입된 서비스, 리포지토리 등)까지 훑는다.
- 대상이 모호하면(예: "이거 설명해줘"인데 파일도 함수도 특정되지 않음) 무엇을 말하는지 되묻는다. 추측으로 엉뚱한 코드를 설명하지 않는다.

이해가 부족한 채로 요약하지 말 것. 호출 관계나 데이터 흐름이 불확실하면 관련 파일을 몇 개 더 읽어 확인한다. 확인되지 않은 동작은 단정하지 말고 "~로 보인다"로 표기한다.

## 출력 형식

항상 아래 4개 섹션을 이 순서로 출력한다. 섹션 제목도 그대로 사용한다.

```
## 한줄 요약
(이 코드가 무엇인지 한 문장. 역할과 소속 도메인이 드러나게.)

## 로직 흐름
(입력 → 처리 → 출력을 단계별로. 번호 목록으로. 분기·예외·비동기 지점을 명시.)

## 주의점
(호출자가 알아야 할 함정: 부작용, 트랜잭션 경계, null·예외 처리, 동시성,
 외부 의존(DB·Kafka·네트워크), 인증·권한 전제 등. 없으면 "특이한 함정 없음"이라고 쓴다.)

## 개선점
(우선순위 순으로. 각 항목은 [무엇을] — [왜] 형태로. 근거 없는 취향성 제안 금지.)
```

## 각 섹션 작성 지침

**한줄 요약** — 무엇을 하는 코드인지 한 문장으로. 이 프로젝트는 도메인이 나뉘어 있으므로(auth, user, device, sensordata, alert, global) 어느 도메인의 무슨 책임인지 드러나면 좋다. 예: "센서 데이터를 Kafka 토픽으로 발행하는 Producer".

**로직 흐름** — 실제 코드가 하는 일을 순서대로. 추상적 재서술이 아니라 코드를 따라간다. 조건 분기, 조기 반환(early return), 던지는 예외, 비동기(@KafkaListener, @Async 등) 경계, 트랜잭션 시작/커밋 지점을 빠뜨리지 않는다. 호출하는 협력 객체가 무슨 일을 하는지도 한 줄씩 짚는다.

**주의점** — 이 코드를 쓰거나 고칠 사람이 모르면 사고 날 지점. 다음을 점검:
- 부작용: DB 쓰기, 메시지 발행, 외부 호출, 상태 변경
- 트랜잭션: 경계가 어디인지, 롤백 조건, 지연 로딩(LazyInitialization) 위험
- 실패 모드: null 가능성, 잡히지 않는 예외, 부분 실패(예: 저장은 됐는데 알림 발행 실패)
- 동시성·순서: 동시 호출 시 문제, Kafka 소비 순서 가정
- 전제 조건: 인증/권한이 이미 통과됐다는 가정, 특정 프로파일에서만 동작(예: Kafka는 prod에서 비활성)
- 실제로 걸리는 게 없으면 억지로 만들지 말고 "특이한 함정 없음"으로.

**개선점** — 실행 가능하고 근거 있는 것만, 영향 큰 순서로. 형식은 `무엇을 — 왜`. 예: "Producer 발행 실패 시 예외를 삼키지 않고 처리 — 지금은 실패해도 호출자가 성공으로 오인함". 프로젝트 가이드상 요청 없는 리팩터링은 하지 않으므로, 여기서는 **제안만** 하고 코드를 고치지 않는다. 사용자가 이후에 고쳐달라고 하면 그때 수정한다.

## 규모에 맞추기

- 짧은 유틸 함수 하나면 각 섹션이 1~2줄로 충분하다. 억지로 늘리지 않는다.
- 큰 파일(여러 책임을 가진 컨트롤러·서비스)이면 로직 흐름을 public 메서드 단위로 나눠 설명한다.
- 개선점이 정말 없으면 "현재 뚜렷한 개선점 없음"이라고 솔직히 쓴다. 채우기용 제안 금지.

## 언어

설명은 한국어로 작성한다. 코드 식별자·타입명은 원문 그대로 둔다.
